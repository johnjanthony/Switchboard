## Ordering decision (server-first)

- **Chunk 2 - agent path:** `join_conversation` + structured JSON returns + the single SKILL.md protocol revision. Server-only.
- **Chunk 3 - human path:** convene command + dispatcher + wake delivery + Operator multi-select. Live sessions only.
- **Chunk 4 - Android batch:** Sessions board, spawn resume-picker rework, phone convene multi-select, resume-into-conversation. One Gradle/deploy cycle.
- **Chunk 5 - deprecation:** open/enter retired and the open marker + lobby machinery deleted; SKILL.md final trim. (Originally sketched as a two-version shim window; collapsed to outright deletion, decided 2026-07-07 - see the chunk 5 section.)

Rationale: convening via Operator needs no Android work (multi-select rides on chunk 1's Sessions rail), and during development convening is as much an at-desk operation as a phone one - so the phone board is off the critical path. Server momentum continues directly from chunk 1, convening becomes usable earliest, and all slow-iteration Android work batches into one chunk. Rejected orderings: phone-first (roster on the phone soonest, but convening lands last and Android is touched twice) and convene-first (human path before agent path splits the SKILL.md protocol churn across two windows).

## Chunk 2 - `join_conversation` + structured returns

**The tool:** `join_conversation(sender, ref?)`. Never blocks.

- `ref` absent: if the open marker points at an active conversation, join it; otherwise mint a new conversation, **promote it to the open marker**, and return immediately. The promotion is what preserves today's pairing semantics through the deprecation window: the second ref-absent joiner lands with the first instead of minting a second room. Waiting for a peer stops being implicit in the join: a joiner who wants to wait calls `message_and_await_agent` next (already the blocking verb).
- `ref` given (a conversation_id from `lookup_conversation_ids`, a convene notice, or John's prompt): idempotent join. Already a member → `{"status":"ok","already_member":true}`. Bound to a different conversation → migrate (same explicit-move semantics as enter's Branch 3 today).
- **History is synchronous:** the join response carries the conversation log that `_queue_for_intro` delivers today by blocking. No intro-queue wait exists on the join path.

**Structured returns:** `join_conversation`, `message_and_await_agent`, `leave_conversation`, `combine_conversations`, and `lookup_conversation_ids` return one-line JSON with a `status` field: `ok | timeout | conversation_ended | conversation_empty`, plus `conversation_id`, `peers`, `log`, `cause` as applicable. `ask_human` keeps its bare-string reply for human answers; only its terminal sentinels adopt the JSON shape (`{"status":"timeout"}`, `{"status":"conversation_ended","cause":"force-ended"}`), so sentinel string-matching leaves agent-land in one revision.

**SKILL.md:** rewritten once, in this chunk, documenting only the new surface. open/enter keep working, undocumented, until chunk 5. One plugin version bump covers the SKILL revision.

**Not touched in chunk 2:** open/enter handlers, the open marker, `open_peer_future`, the lobby-hold branches (all die in chunk 5). The wait-queue/talking-stick core is untouched throughout.

**As-built deltas (landed 2026-07-06):** `join_conversation` also takes optional `title` (used on the mint path) and an `already_member` re-join returns the UNSEEN delta in `log` (join is uniformly "ensure membership + collect unseen history" - the combine-intro flow depends on it); combine's envelope carries `detail` (the server's summary string) instead of a parsed count; `message_and_await_agent`'s "listen without speaking" ERROR string now names `join_conversation()`; the sole-alive lobby-hold is documented in SKILL.md's new-surface terms (wake payload is the JOIN NOTICE, the peer's first message arrives on the next `message_and_await_agent` cycle).

## Chunk 3 - convene command, wake delivery, Operator multi-select

**Command tree:** `convene_commands/<push-id>` - top-level, matching `spawn_commands`/`combine_commands`/`away_mode_commands`. (Delta from the parent spec's `commands/convene/` nesting, for consistency with the existing trees.) Shape: `{session_ids: [...], target: "new" | "<conversation_id>", title, issued_at, status}`. New `dispatch_convene_commands` loop, freshness-gated via `command_freshness.py`, supervised like its siblings.

**Per-session routing** (existing `conversation_ops` machinery): unbound → `_add_member` + bind; bound to a solo conversation → `_migrate_member`; bound to a multi-party conversation → **skipped**, reason recorded on the command result. Pulling a member out of an active collab remains `combine_conversations`' job. A system intro message ("John convened: Claude Win, Claude WSL") lands in the target conversation.

**Wake matrix implementation** (decided: extend existing hooks; no new hook registrations):

| Session state | Mechanism | Latency |
| :--- | :--- | :--- |
| `awaiting_agent` | server resolves the blocked `message_and_await_agent` future with `{"status":"convened", conversation_id, peers, log}` | immediate |
| `awaiting_human` | never preempted; convene notice attached to the pending record and prepended to the eventual answer payload | on answer |
| `active` | per-session **notice queue** on the SessionRegistry (RTDB-mirrored; survives restart). Turn-end hook's GET gains `session_id`; the response gains `notices`; the Stop hook blocks with the notice text as reason - the away-mode enforcement mechanism reused | end of turn |
| `idle` | same queue, delivered as `additionalContext` on the next UserPromptSubmit | John's next touch (passive only in this chunk) |
| `ended` / `lost` | not selectable in chunk 3; resume-into-conversation ships in chunk 4 | - |

Notices pop on read (at-most-once). Backstop for a discarded hook response: the convene intro message already sits in the conversation, so the session learns on its next conversation touch regardless.

**Wake-path derivation rule (2026-07-06 herdr review):** the dispatcher derives each session's wake path from the registry's *live* state at command-execution time, never from the surface's roster snapshot at tap time; and an already-satisfied condition (the session is already a member, already awake, already in the target conversation) resolves as immediate success rather than parking a wake that can never fire (herdr's initial-state-probe pattern).

**Operator:** chunk 1's Sessions rail gains multi-select checkboxes on live rows (`active`/`idle`/`awaiting_*`), a Convene button, a target picker (new conversation + title, or an existing active conversation), and per-row wake-path hints. Writes go through a `commands.js` builder; `node --test` units for the builder and any new derivations.

**Roster polish (added 2026-07-06, John's request):**

1. **Forward `cwd` on agent-status events.** Claude Code's hook stdin already carries `cwd`; the agent-status script just does not send it. Add it to the POST body and have the server fill a record's empty cwd from it. Kills most "(unknown)" rows (sessions discovered via agent-status or rings before their SessionStart existed). Rides the same plugin version bump as this chunk's notice work.
2. **Display the Claude Code session name.** The /rename title is not visible to the server today: hook payloads do not include it and LocalSystem cannot read `~/.claude` (D6). **Empirical gates closed (verified 2026-07-06):** /rename appends a `{"type":"custom-title","sessionId":"<id>","customTitle":"<name>"}` record to the session's own transcript JSONL (`~/.claude/projects/<proj>/<session-id>.jsonl`), and CC also continuously appends `{"type":"ai-title","sessionId":"<id>","aiTitle":"<generated title>"}` records - the auto-titles the `/resume` picker shows, regenerated as the session evolves (ubiquitous: ~1900 records across John's projects; nearly every session has one, vs. custom-title in a single session). Last record of each type wins; custom-title outranks ai-title. Sensor decision: **Watchtower** - it already parses exactly these transcripts for rings, so it extracts both record types and adds `name` (and its source) to the ring payload alongside `model`/`pct`; the server's `apply_rings` enrichment and `_RING_FIELDS` gain the field, and `SessionRecord` gains `name`. Because ai-title regenerates mid-session, the roster label doubles as a living "what is this session doing" description. (Hook-based sensing was the alternative; rejected because it would add per-turn transcript tailing that Watchtower already does. Note: ai-title does NOT flow into the OSC tab title, so it aids roster display, not the backlog heartbeat's tab correlation.)
3. **Rail label chain becomes** custom-title → latest ai-title → sender → cwd tail → "(unknown)". With ai-title ubiquitous, nearly every row gets a meaningful label; `sender` and cwd are the residual fallbacks for brand-new sessions whose first ai-title has not been generated yet.
4. **Needs-attention bit (2026-07-06 herdr review).** herdr's "done" state is not a state-machine state: it is `idle` plus "not acknowledged since the last completion", auto-cleared when the human views the agent. Adopt the same: an `acknowledged` flag on the roster row (flipped by Operator selection / phone open), rendering `idle && !acknowledged` as "done - needs you". No new registry state; mirrors the phone's existing unseen-activity dot for conversations.
5. **State provenance.** `SessionRecord` gains `last_transition_source` (which event set the current state, and when), surfaced in `GET /sessions` and as a row tooltip. The one-field version of herdr's `agent explain`; makes "why does the rail say awaiting_human" answerable at a glance.

**As-built deltas (landed 2026-07-07, `9b83580`):** command entries carry no `status` field - the shared listener deletes commands after dispatch, so outcomes live in the target conversation's intro message plus a phone notice for the all-skipped case; wake dispatch branches on OBSERVED blocking structures (wait-queue entries, `open_peer_future`, pending asks), with the registry state string feeding only the UI's wake-path hints; the multi-party skip rule counts ALIVE members only (a conversation of dormant peers is solo for migration); `session_acks/<id>` is a sibling RTDB tree, not a record field (the mirror rewrites `sessions/<id>` wholesale and would clobber a client-written field), and pruning a session record also deletes its ack.

**Smoke findings (2026-07-07):** live convene verified end to end - idle-path notice arrived as UserPromptSubmit context, `join_conversation(ref)` returned `already_member` with the intro as the unseen log, the peer round-trip rode the normal talking-stick, and the audit log recorded `convene_command_handled` with zero skips. Label-chain coverage note: some sessions never emit ai-title/custom-title transcript records at all (observed live: two VSCode-hosted sessions in the same repo, one emitting, one not - the non-emitter a Fable/Agent-SDK session; cause not isolated), so the `sender`/cwd-tail fallbacks are load-bearing, and after a convene such sessions display the server-stamped `Agent <first-8>` sender. Chunk 4's phone label chain inherits the same boundary.

**Parent open questions, resolved:**

1. *CLI-agnostic convene notices* - scoped to Claude for now. The turn-end hook's `--cli` abstraction stays; Gemini verification is deferred (the repo already de-emphasized Gemini naming).
2. *Force-pull from multi-party conversations* - no. Skip-and-report; combine is the forceful verb. Revisit only if real usage shows repeated convene-then-combine.
3. *Default-room semantics after the marker retires* - keep join-or-create for `ref`-absent joins; it remains the zero-configuration path for ad-hoc two-agent work.

## Chunk 4 - Android batch + server resume machinery (designed 2026-07-06)

**Scope:** one Gradle/deploy cycle for the phone work, plus the server-side resume machinery the phone actions require, plus a small Operator ride-along (decided 2026-07-06: ride along rather than defer). Brainstormed with mockups 2026-07-06; all decisions below John-approved. Delta from this doc's original sketch: the spawn "+" sheet does NOT gain a resume mode - resume lives on the Sessions board and the conversation-row shortcut instead.

**Sessions board (phone).**

- New `"sessions"` NavHost route (4th route). Entry: a top-bar icon on the conversation list (between "+" and overflow) with a badge counting needs-attention sessions (`idle` and unacknowledged); no badge at zero. Board top bar: back / "Sessions" / AwayModePillChip.
- One scrollable list, two groups: LIVE (`active | idle | awaiting_human | awaiting_agent`; needs-attention rows first, then `last_event_at` desc) and a collapsed-by-default "RECENTLY ENDED (n)" expander (`ended` + `lost`, `last_event_at` desc). `lost` renders with its own chip but is treated as ended - the underlying CC session is often still resumable.
- Row anatomy (Operator-rail parity, the "rich" option): status lamp; line 1 = name via the label chain (custom-title -> ai-title -> `sender` -> cwd tail -> "(unknown)"); line 2 = project tail, surface, state detail ("tool:Bash", "asked you a question", "turn ended"); line 3 (bound rows only) = a tappable `<-> <conversation title>` chip. Right column: state chip, context-% badge (only >50%, the existing ContextBadge rule, fed from `context_pct`), relative time.
- Interactions (conversation-first): tap = write `session_acks/<id>` + deep-link to the bound conversation's chat page; unbound rows open the session detail sheet instead. Long-press = dropdown menu (the conversation-row idiom): "Convene into..." (live rows; enters selection mode preselected), "Details", "Resume..." (ended/lost rows). The detail sheet is the provenance surface - name + source, state + `last_transition_source`, model, full cwd, surface, session id, started/last-event times, plus the row's actions - and opening it also acks.

**Convene multi-select (phone).** Contextual action bar selection mode ("N selected" / cancel / CONVENE) with check circles on selectable rows; per-row wake-path labels while selecting ("wakes instantly" / "on next phone answer" / "at end of current turn" / "on your next prompt"); ended/lost rows selectable, labeled "Resume into conversation" (the fork case). CONVENE opens a bottom sheet: radio "New conversation" + optional title, or an existing active conversation. Confirm writes ONE `convene_commands` push in chunk 3's exact shape - the phone is just a second writer of the same command.

**Resume (phone).**

- Board ended/lost row -> "Resume..." sheet: optional prompt + target radio: **Standalone** (default; no conversation - the session-fallback machinery mints a home conversation when the agent first needs John), **Back into the old conversation** (offered only when the record's `conversation_id` still points at an Active conversation, labeled with its title), or **Into existing...** (picker over active conversations).
- Conversation-row long-press Resume stays, with today's continuation semantics (new conversation, `continued_from`), but its enablement switches from member archaeology to "any member session id has an `ended`/`lost` registry record". `isResumable` member-scanning and `staleSessionWarning` are deleted from the Android models.
- **Resume window = registry retention (72h default), accepted:** sessions older than that are not resumable from the phone (the terminal still works); `SWITCHBOARD_SESSION_RETENTION_HOURS` already exists if it ever pinches. No archaeology fallback - one mechanism.

**Server.**

- Command shapes: `convene_commands` unchanged. Conversation-row resume keeps `{type:"resume", source_conversation_id, prompt?}`. NEW `{type:"resume_session", session_id, target_conversation_id?, prompt?, issued_at}` for board resumes; absent target = standalone.
- `handle_resume_session` (new SpawnHandler entry point): registry record must be `ended`/`lost`; same `quser` interactive-session gate; launches `claude --resume <session_id>` in the record's cwd/surface via the landed pending-file + `spawn-launcher.ps1` mechanism (the `combine_resume` template - the launcher already maps resume types to `--resume`). Prompt by target: standalone -> come online via ask_human/notify_human (the T-147 solo rule); into-conversation -> `join_conversation(sender, ref='<conv-id>')` to collect history, plus a system message in the target (`John resumed <name> into this conversation`).
- **Session ids are stable across resume (verified live 2026-07-07):** a headless `claude -p --resume <old-id>` on the installed CC appended to the SAME transcript with the SAME sessionId - no fork, no new file. Consequences: the returning session revives its existing registry record via SessionStart under the same id; into-conversation resume pre-adds the member at SPAWN time under that stable id (exactly like `handle_resume`'s landed pre-bind, which is CORRECT as-is - the ghost-member wart previously claimed in this section was wrong and is retracted); standalone resume needs zero conversation work (the first `ask_human` mints a home conversation via the landed `mint_if_unbound` machinery). The supersession ledger / `superseded_by` / tombstone / rekey machinery designed for an id-change world is CUT from this chunk (decided 2026-07-07) - see the dormant contingency below.
- **Id-change sentinel (the one survivor of the cut):** each spawn-driven resume records `{session_id, cwd, issued_at}` in a small in-memory recent-resumes list; a `/session_start` with `source="resume"` whose session_id has NO existing registry record while a recent resume targeted a DIFFERENT id in the same cwd logs a loud JSONL line (`resume_id_change_detected`). If CC ever changes id-on-resume behavior, it announces itself instead of silently breaking membership.
- **Away-mode rule (decided):** fresh spawn and both resume paths keep/gain the auto-enable; convene NEVER touches away mode, including its fork-arm - an at-desk Operator convene that sweeps in an ended session must not flip John to away.
- Convene fork-arm: `_perform_convene` stops skipping terminal sessions - an `ended`/`lost` id routes to the resume machinery with target = the convene target and the convene notice as its prompt (member pre-added at spawn time, stable id); the intro message lists them ("John convened: X, Y; resuming: Z"); no wake path applies (the spawn IS the wake).

**Resume supersession rules (2026-07-06 herdr review; DORMANT contingency as of 2026-07-07):** honor a session-id change only when it arrives with a recognized lifecycle reason - the SessionStart hook already carries `source` (`startup|resume|clear|compact`). Verified 2026-07-07: the installed CC preserves session ids across `--resume`, so no flow in this program triggers an id change and nothing implements these rules; the id-change sentinel above is the tripwire. If a future CC version forks ids on resume, this paragraph is the design to implement: link old to new (`superseded_by` on the old record) at launch, tombstone the old id (straggler events ignored rather than flapping the superseded record back to `active`), and rekey conversation membership; herdr's stale-session set is the reference mechanism.

**Operator ride-along.** `derive.isConvenable` widens to `ended`/`lost`; `wakePathHint` gains ended/lost -> "Resume into conversation"; rail ended rows get selection checkboxes. Also fixes a landed display bug: the rail renders `Math.round(record.context_pct)%` but `context_pct` is 0-1 scale (every row shows 0% or 1%) - multiply by 100. Node tests extend chunk 3's.

**Android data layer.** `RegistrySession` data class in the shared module's `network/Models.kt` (`@PropertyName` snake_case, mirroring `SessionRecord`); `sessions/` and `session_acks/` listeners plus writers `ackSession` / `conveneSessions` / `resumeSession` in the shared `MainViewModel`. All derivations in a new pure `SessionBoardPolicy.kt` beside `ConversationPolicy.kt` - label chain, needsAttention (mind chunk 3's `+00:00`-vs-`Z` timestamp-parse gotcha), wake hints, live/ended partition + sort, badge count - unit-tested in `shared/src/test/`. New composables with collision-safe names: `SessionsBoardScreen`, `RegistrySessionRow`, `SessionDetailSheet`, `ConveneTargetSheet`, `ResumeSessionSheet`. **Legacy rename (approved):** `SessionListScreen` / `SessionViewScreen` / `SessionRowComposable` - which render conversations - become `ConversationListScreen` / `ConversationViewScreen` / `ConversationRowComposable`, killing the terminology trap the board would otherwise worsen. Wear module untouched (shared additions are additive; the wear build must stay green).

**Deploy notes.** RTDB security rules: the phone's Firebase auth needs read on `sessions/` and write on `session_acks/` (verify against the grants backing Operator's rail) - a John-assisted console step alongside `install-client.ps1`. No hook or plugin changes (SessionStart already carries `source`), so **no plugin version bump this chunk**.

**Non-goals.** No Wear UI; no new FCM channels or notifications (needs-attention stays passive on the board); T-001 parked pendings untouched; open/enter deprecation stays chunk 5.

**As-built deltas (landed 2026-07-07, `e2a5e0e` + `06790e1`; smoke passed same day):** `RegistrySession` omits `pending_notices` (the board never renders queued notices; `@IgnoreExtraProperties` absorbs it). ## Chunk 5 - legacy deletion (designed 2026-07-07; supersedes the original shim-window sketch)

Brainstormed 2026-07-07 while chunk 4 executed; decisions John-approved. The original two-plugin-version shim window is dropped: every caller is John's own agent fleet and the spawn prompts have said join/ask since chunk 2, so the tools are deleted outright.

- **Delete outright, no shims (decided):** `open_conversation` / `enter_conversation` tool registrations, handlers, and SKILL.md paragraphs are removed in one chunk. An audit task greps spawn prompts, SKILL.md, and docs for straggler references before the deletion lands. A long-lived session with a cached old SKILL that still calls them gets a tool-not-found error and adapts; accepted.
- **Machinery deleted with them:** `open_peer_future`, `_queue_for_open_peer`'s opener arm, `message_and_await_agent`'s lobby-hold branch, chunk 3's `_convene_lobby_futures` handoff, and the singleton open marker. The server stops maintaining `open_conversation_id` and nulls the RTDB path once at startup; the phone's open-accent UI goes INERT (listener reads null, accent never renders) rather than being removed - chunk 5 needs no Gradle/deploy cycle, and the dead Android code is swept in chunk 6 (the next Android-touching chunk).
- **Ref-less join, the marker's replacement (decided):** `Conversation` gains an `origin` field (`join | spawn | convene | fallback | resume`), persisted in conversation meta and hydrated; pre-existing records have no origin and are never candidates. A ref-less join targets the candidate iff EXACTLY ONE exists: Active, `origin == "join"`, exactly one alive member, created within ~30 minutes; otherwise mint with `origin="join"`. (Resolved open question 3 stands: join-or-create stays.) Deliberate behavior change from marker days: a THIRD ref-less joiner mints its own room - the pair stops being a candidate once both members are alive; three-way setups use refs or convene.
- **Solo wait parking (decided; replaces the parent spec's uniform `conversation_empty`):** a sole-alive `message_and_await_agent` appends its message and parks in the NORMAL `wait_queue` - a later ref-less joiner reads the message on join and their reply wakes it through the ordinary talking-stick; convene's awaiting_agent instant-wake and the existing wait timeout apply unchanged. Two-terminal zero-config pairing therefore keeps its wait step end to end. `conversation_empty` loses its last producer and RETIRES from the envelope vocabulary (leave still wakes parked waiters with the parting message; force-end still resolves `conversation_ended`). The resume/spawn prompt lines that warn solo agents away from blocking (T-147 phrasing, chunk 4's `_format_resume_session_prompt` solo arm) are softened to match.
- SKILL.md loses the last legacy paragraphs and the `conversation_empty` row; one plugin version bump.
- **Cancelled-`message_and_await` session-state reset (found in chunk 2 smoke, 2026-07-06).** Cancelling a `message_and_await_agent` from the CLI cleans up the in-memory wait future correctly - no leaked waiter - but nothing resets the caller's session state from `awaiting_agent` back to idle, so the Operator roster shows a stale "waiting on agent" chip until the next hook event reconciles it. Fix: reset session state in the handler's `CancelledError` path (only the normal wait branch remains once the lobby branch is deleted). Display-staleness only; low severity.
- T-001's parked-pendings chunk follows as chunk 7, after the chunk 6 sensing batch - designed 2026-07-07, see `2026-07-07-t001-parked-pendings-design.md` (scope verdict in the re-triage decision doc).

**As-built deltas (landed 2026-07-07, `0580778`, plugin 1.4.0; smoke passed same day):** `ask_human`'s CancelledError arm got the same session-state reset as `message_and_await_agent`'s (identical gap in the sibling handler, same two lines). The first-MCP-call ad-hoc mint stamps `origin="fallback"` (no sixth origin value; the candidate rule only needs "not join"). `_wake_convened` kept only its wait-queue / pending-notice / queue_notice arms - the lobby arms lost their sole producer and were deleted with it. Two deletion sites the plan's grounding missed were found and handled by the implementer: an `open_peer_future` dormancy-resolve in `cli_session_end.py` and the `messenger.py` protocol entries. Operator's marker subscription and `" open"` row rendering were removed in-chunk (zero-build; the phone's open-accent goes inert by data as designed, its dead code swept in chunk 6). One execution-time misgrounding (an orchestrator brief overrode the plan's correct `self._now()` to the module `_now_iso()`) was caught by John post-review and reverted with a frozen-clock regression test; the lesson (verify `self.x =` assignments, not just `def x`, before declaring a symbol absent) is recorded in project memory.

## Chunk 6 - sensing (promoted from Backlog, designed 2026-07-07; T-001 becomes chunk 7)

**Decisions (2026-07-07) on top of the verified backlog material below:**

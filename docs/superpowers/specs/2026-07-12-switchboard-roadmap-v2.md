- **T-181 — register-first (decided 2026-07-15).** `registry.add` moves BEFORE the question write in `ask_human`: mint `request_id`, register with `msg_id=None` (the field is already Optional and `add` is fully synchronous), then run the write and patch `record.msg_id` from its return. This closes the race to zero width by construction — there is no await between the away-gate check and registration, so an answer cannot be processed before the correlation exists. The two rejected shapes: splitting the write into push + finish-delivery phases (new backend surface across the messenger ABC and every fake backend, and a residual window remains — the answers-listener callback can be queued ahead of the ask's post-push resume), and dispatch-side buffering of unknown-correlation answers (complexity in the dispatch loop, delayed truth, and the registration gap survives for every other consumer). Consequences banked: (a) the REV-002 residual re-check (WP-2) is KEPT but its rationale is rewritten — with register-first the away-exit drain always snapshots this pending, so the re-check now covers only the drain-crash path (`_apply_bulk_respond_decision` raising mid-drain) and future re-orderings; its semantics (withdraw + at-desk sentinel, fall-through preserves a drain-delivered bulk reply) are unchanged. (b) The write-failure arm gains compensating cleanup through `terminate_pending` (single-terminal-owner invariant preserved; `mark_question_cancelled`'s message walk no-ops safely when the question message never landed). (c) The `add_pending_question_record` spawn is gated on `not future.done()` — an ask already answered mid-write, superseded, or drain-resolved never spawns the phone record, which also tightens the pre-existing supersede-vs-record race. (d) Accepted cosmetic: an answer landing inside the write window delivers with `attached_to_msg_id=None` (reply not spliced under its question bubble); the pending-mirror count also increments a moment before the question is phone-visible. (e) The superseded-prior cancel stays positioned after the write (unchanged relative order; keeps the supersede-cancel race tests' shape).
- **T-185 — optimistic flip + rollback-to-dormant (decided 2026-07-15).** The pre-launcher flip stays (it is deliberate: the bound=alive-or-launching invariant keeps `message_and_await_agent`'s alive-peer count honest and closes the double-launch window a dormant-while-launching member would open to a concurrent phone Resume). What changes: when `invoke_spawn_launcher` fails, every member flipped for that launch is rolled back under the conversation lock — captured pre-flip dormancy fields restored, `alive=False`, session unbound (dormant members must be unbound per the WP-3 invariant; `handle_resume`'s eligibility filter logs drift otherwise) — and the dormant state is re-persisted with an AWAITED `write_conversation_member` (not `_spawn_bg`; the awaited write is ordered after the optimistic persist, which completed during the launcher's subprocess round-trip), so restart+hydration sees dormant and the backlog's persistence complaint is fixed. Confirm-then-flip was rejected: it matches WP-4's convene reserve-then-materialize precedent and has a nicer crash story, but it opens the dormant-while-launching window (double-launch, alive-peer dips, first-call-before-bind), restructures both flows' happy paths, and `schtasks` success is the same weak signal either way (T-142's H08/H09 class). Scope covers THREE sites: `_perform_combine`'s dormant arm, `handle_resume`'s flip loop, and the site the backlog missed — `handle_resume_session` pre-adds an alive member before `launch_resume_agent`; on `ok=False` a member added by that call is marked dormant (`launch-failed` reason) + unbound + persisted, preserving board-resume retryability (`note_spawn_resume` is sentinel-only; the registry record stays terminal). Failure notice texts are rewritten to advertise the recovery that now works (phone Resume long-press on the target/new conversation). The Antigravity guards stay untouched and fire before any flip/rollback logic (an agy member is never flipped, so never rolled back). Accepted residuals, same class as the parked T-142: a process crash between the optimistic flip-persist and the rollback still strands (window is the launcher subprocess call), and T-142's gate TOCTOU is neither widened nor fixed.
